Duplicate Asset Detection
Card Tile Decoration

In AEMaaCS, binary uploads bypass the JVM via Direct Binary Access to Azure Blob Storage, making duplicate detection an async post-processing concern handled by NUI. Today the detection result surfaces only as an inbox notification — never on the asset card itself. This document maps the current architecture, identifies the gap, and outlines a targeted co-innovation approach.

Current State — Async Detection Flow

How duplicate detection works today

Unlike traditional AEM where the binary streams through the JVM enabling synchronous SHA1 comparison, AEMaaCS uploads bypass Java entirely. The duplicate check runs post-processing via AssetProcessorEventHandler, triggered after NUI finishes generating renditions and writing dam:sha1 to jcr:content/metadata.

The Gap

Detection result never reaches the card tile

⚠

Notification-only output — no JCR state written

AssetDuplicationDetector.duplicationDetect() calls TaskManager.createTask() to fire an inbox notification, then exits. It does not write any property back to jcr:content/metadata. As a result, card-banner.jsp has no data to read and the card renders without any duplicate indicator — the user must navigate to their inbox to discover the issue, often hours after upload.

What exists today

AssetDuplicationDetector.java

Runs post-NUI. Performs a QueryBuilder search by dam:sha1. On match: calls sendNotification() which calls TaskManager.createTask("Notification") with a text description. No metadata written. No JCR state change. Feature flag defaults to enabled=false via OSGi config.

What's missing

card-banner.jsp — no duplicate path

The banner JSP reads asset metadata, dam:assetState, workflow status, and Scene7 flags to decide which badge or alert to render. It renders a blue "New" tag, a "Licensed" tag for Stock, and various <coral-alert> states — but has no branch for duplicate status. There is no property to read because nothing writes one.

Co-Innovation Approach

Two-layer change: persist state, then surface it

→

Core idea

When duplicationDetect() finds a match, also write a new metadata property dam:duplicateAssets (multi-value String) back to jcr:content/metadata of the newly uploaded asset, containing the paths of its duplicates. card-banner.jsp can then read this property and render an amber <coral-tag color="orange">Duplicate</coral-tag> — using exactly the same badge pattern as "New" and "Licensed".

Step 1 — Backend

Write metadata on detection

In AssetDuplicationDetector.duplicationDetect(), after building dupAssetsMap, open a service ResourceResolver and write dam:duplicateAssets (String[]) to jcr:content/metadata of each flagged asset. Then call resolver.commit() before the existing notification logic. Notification is preserved — this is additive.

Step 2 — JCR Schema

New metadata property

Property: dam:duplicateAssets
Type: String[] (multi-value)
Node: jcr:content/metadata
Value: absolute DAM paths of duplicates

Using a multi-value string array allows one asset to declare multiple duplicates, and the list can be updated if more duplicates surface later. No new node type required — fits existing metadata schema.

Step 3 — Frontend

card-banner.jsp badge branch

Read dam:duplicateAssets from the metadata ValueMap. Insert a new else if (isDuplicate) branch in the badge decision chain — after the isNew branch — that builds an AttrBuilder with color="orange" and renders <coral-tag>Duplicate</coral-tag>. Matches the existing "New" / "Licensed" pattern exactly.

JCR Schema

Proposed metadata property

jcr:content/metadata — new property
Property name JCR type Description
dam:duplicateAssets String[] Multi-value. Each value is the absolute JCR path of a known duplicate (e.g. /content/dam/folder/original.jpg). Written by AssetDuplicationDetector post-processing. Absent means no duplicates detected yet.
dam:duplicateDetectedAt Date Optional. Timestamp of when the detection ran. Useful for cache invalidation and stale-state debugging.

Code Sketch

Key implementation lines

AssetDuplicationDetector.java Java — duplicationDetect() 
// After dupAssetsMap is populated...
if (!dupAssetsMap.isEmpty()) {

    // NEW: persist duplicate state to metadata
  for (Map.Entry<String,String> dup : dupAssetsMap.entrySet()) {
    Resource meta = resolver.getResource(
      dup.getKey() + "/" + JCR_CONTENT + "/" + METADATA_FOLDER);
    if (meta != null) {
      ModifiableValueMap mvm = meta.adaptTo(ModifiableValueMap.class);
      mvm.put("dam:duplicateAssets", dup.getValue().split(","));
    }
  }
  resolver.commit(); // flush JCR write

  // existing: send inbox notification
  String userId = getUserId(resolver, detectionAssetList.get(0));
  sendNotification(resolver, userId, description.toString());
}
card-banner.jsp JSP — badge decision chain 
// At top: read duplicate state from metadata
ValueMap cardMeta = resource.getChild(
    JCR_CONTENT + "/" + METADATA_FOLDER).getValueMap();
String[] dupPaths = cardMeta.get(
    "dam:duplicateAssets", String[].class);
boolean isDuplicate = dupPaths != null && dupPaths.length > 0;

// In badge decision chain (after isNew branch):
} else if (isNew && ...) {
  // existing New tag logic
} else if (isDuplicate) {
  attrDuplicateTag = new AttrBuilder(request, xssAPI);
  attrDuplicateTag.add("color", "orange");
  attrDuplicateTag.add("class", "u-coral-pullRight");
  hasAlert = false;
}

  </coral-tag>
<% } %>

Open Questions

Areas to explore further

Q1 — Clearing the flag

When does the badge disappear?

If the user deletes the original asset, should dam:duplicateAssets be cleared? Needs an event handler on dam:assetDeleted or a periodic revalidation job. Stale badges erode trust.

Q2 — Bidirectionality

Which asset is "the duplicate"?

Today the newer asset (the one being processed) gets flagged. Should the older asset also get a "Has duplicates" badge? Symmetric flagging changes the UX story for the co-innovation pitch.

Q3 — Click behavior

What happens on badge click?

The "New" and "Licensed" tags are purely decorative. A "Duplicate" badge that links to the original asset (using dam:duplicateAssets[0]) would be significantly more actionable — but requires client-side JS on the card component.
